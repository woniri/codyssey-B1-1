# 시스템 설계 문서 비교 평가 — v1 → v2 단계적 추론 개선

## 1. 테스트 목적

이번 테스트는 회의록 요약 AI가 정보가 부족한 상황에서 어떻게 대응하는지 비교하기 위한 테스트.

특히 단순히 회의록을 작성하는 능력보다, 다음 능력을 중점적으로 평가.

* 제공된 정보의 부족함을 인식하는가
* 추측하지 않고 확인 질문을 하는가
* 누락 정보를 항목별로 정리하는가
* 사용자가 다음에 무엇을 제공해야 하는지 명확히 안내하는가
* 최종 회의록 작성 전 검증 단계를 거치는가

## 3. 결과별 평가

### GPT 결과 평가

#### 장점

* 결정사항, Action Items, 리스크, 후속 일정 형식을 유지했함,정보 부족을 안내함.
* 추가 확인 요청 사항을 비교적 자세히 제시.

#### 단점

* 제공된 정보가 부족한데 먼저 회의록 형태로 작성함.
* “프로젝트 전체 리스크가 발생할 가능성”, “동일한 이슈가 재발하지 않도록 관리 필요” 등 일부 추론 포함.
* Action Items가 비어 있는 상태라 실제 업무 문서로 쓰기 어려움.

#### 총평


v1은 부족한 정보를 기반으로 회의록을 먼저 생성하려는 경향이 있었으며,
일부 추론이 포함될 가능성이 존재하였다.

반면 v2는 정보 부족 여부를 먼저 진단하고,
핵심 누락 정보를 질문을 통해 보완한 후
최종 회의록을 생성하도록 개선되었다.

특히 정보가 불충분한 상황에서 환각(Hallucination)을 줄이고
실무 활용성을 높이는 데 효과적임을 확인하였다.---

### Claude 결과 평가

#### 장점

* 정보 부족을 명확히 인식함,메모 공유를 요청한 점이 좋음, 사용자에게 친절하고 읽기 쉬운 방식으로 질문함
* 추측 없이 추가 정보 확인을 우선함

#### 단점

* 질문 항목이 4개로 압축되어 있어 일부 정보가 빠질 수 있으며, 회의 일시, 참석자, 예산 조정 규모 등 세부 항목이 충분히 분리되지 않았음.

#### 총평

v2 방향으로 약간 개선된 결과. 다만 실무 문서용 시스템 프롬프트로 쓰기에는 질문 구조를 조금 더 정교하게 만들 필요가 있음

---

### 3번 결과 평가

#### 장점

* 정보 부족을 가장 논리적으로 설명함, 원문에서 확인 가능한 표현과 부족한 정보를 구분함.
* 일정, 예산, 의견 대립, 담당자, 후속 일정으로 질문을 잘 분류함, 회의 메모 전문을 요청해 환각 가능성을 낮췄음.

#### 단점

* 사용자가 초보라면 질문량이 조금 부담스러울 수 있음.

#### 총평

“먼저 누락 정보를 진단하고, 그다음 필요한 정보를 요청한다”는 단계적 추론 구조가 잘 드러났음.

---

### 4번 결과 평가

#### 장점

* 가장 안정적인 결과, “주어진 내용만 기반으로 작성한다”는 원칙을 먼저 선언함
* 기본 정보, 일정, 예산, 의견, 후속 조치로 질문 카테고리를 잘 나눴음, 회의록 작성 전에 필요한 정보를 체계적으로 확인했함
* 사용자 친화성과 실무 활용성이 모두 높음.

#### 단점

* 매우 간결한 결과를 원하는 사용자에게는 질문이 많아 보일 수 있다.

## 4. V1 최종 비교 결론

이번 테스트에서는 Grok 결과가 가장 우수함, 회의록을 바로 작성하지 않고, 먼저 정보 부족을 명확히 알린 뒤 필요한 정보를 항목별로 질문해서 “추측하지 말고 주어진 내용만 기반으로 요약한다”는 조건을 가장 잘 지킨 방식.
Gemini 결과도 매우 우수하나  Gork 결과가 사용자 친화성과 문서 구조 면에서 조금 더 안정적임.

GPT는 형식상 회의록을 만들었지만, 정보가 부족한 상황에서 일부 내용을 추론해 작성했음, 따라서 v1에서 v2로 개선할 때는 “일단 작성”하는 방식에서 벗어나, “정보 부족 진단 → 확인 질문 → 메모 요청 → 최종 요약” 순서로 바꾸는 것이 적절함

---

## 5. v1 → v2 개선 방향

### v1 문제점

v1은 정보가 부족해도 회의록 형식을 먼저 채우려는 경향이 있다.

이 경우 다음 문제가 발생한다.

* 원문에 없는 리스크를 만들어낼 수 있다.
* 담당자와 기한이 없는 Action Items를 억지로 구성할 수 있다.
* 사내 공유 문서로 사용할 때 사실과 추측이 섞일 수 있다.
* 사용자는 AI가 작성한 내용이 실제 회의 내용이라고 오해할 수 있다.

---

### v2 개선 방향

v2는 회의록을 작성하기 전에 먼저 정보를 점검한다.

개선된 처리 순서는 다음과 같다.

1. 제공된 회의 내용에서 확실한 정보만 추출한다.
2. 누락된 핵심 정보를 항목별로 분류한다.
3. 정보가 부족하면 회의록을 완성하지 않고 확인 질문을 한다.
4. 사용자가 추가 메모를 제공하면 그때 최종 회의록을 작성한다.
5. 최종 출력에서는 추측과 사실을 섞지 않는다.

---

## 6. V2 평가 및 비교 결론

4개의 모델 모두 확인 질문을 먼저 하는 패턴을 보임, 각 모델간 차이점이 보이지 않음.

## 7. 최종 평가

v1은 부족한 정보를 기반으로 회의록을 먼저 생성하려는 경향이 있었으며,일부 추론이 포함될 가능성이 존재함.
반면 v2는 정보 부족 여부를 먼저 진단하고,핵심 누락 정보를 질문을 통해 보완한 후 최종 회의록을 생성하도록 개선되었음.
특히 정보가 불충분한 상황에서 환각(Hallucination)을 줄이고 실무 활용성을 높이는 데 효과적임을 확인하였음.
앞으로 페르소나, Few-shot, 단계적 추론을 거치면 어떤 모델을 사용해서 원하는 결과를 얻을 수 있을 것 같음.
