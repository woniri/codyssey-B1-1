# 02. 시스템 설계 문서

## 1. 문제 정의

### 1.1 선택한 업무 과업

회의록 요약 자동화

### 1.2 타겟 사용자

* 프로젝트 매니저(PM)
* 팀장
* 실무 담당자
* 회의 결과를 빠르게 공유해야 하는 조직 구성원

### 1.3 업무 문제점

기존 회의록 작성 과정에서는 다음과 같은 문제가 발생한다.

1. 회의 내용이 길고 비정형적으로 기록되어 핵심 내용을 빠르게 파악하기 어렵다.
2. 결정사항과 Action Items가 명확하게 구분되지 않아 후속 업무가 누락될 수 있다.
3. 회의록 작성 및 정리에 많은 시간이 소요된다.
4. 회의 중 오간 의견과 논의 과정이 혼재되어 핵심 결과를 식별하기 어렵다.
5. 담당자, 일정, 리스크 정보가 체계적으로 정리되지 않는 경우가 많다.

### 1.4 목표

* 회의 내용을 빠르고 정확하게 구조화된 형태로 변환한다.
* 결정사항과 Action Items를 명확하게 분리하여 후속 업무의 실행력을 높인다.
* 일관된 출력 형식을 유지하여 가독성을 향상한다.
* 장문의 회의 내용을 효율적으로 요약한다.
* 환각(Hallucination)을 최소화하여 사실 기반 결과를 제공한다.
* 10턴 이상의 대화에서도 문맥을 안정적으로 유지한다.

### 1.5 입력 데이터 형태

* 회의 메모
* 음성 녹취 텍스트
* 메신저 대화 기록
* 회의록 초안
* 자유 형식 텍스트

### 1.6 출력 규격

#### 결정사항

회의에서 최종적으로 합의되거나 결정된 내용

#### Action Items

| 담당  | 기한 | 작업 내용     |
| --- | -- | --------- |
| 역할명 | 일정 | 수행해야 할 업무 |

#### 리스크

프로젝트 또는 업무 진행에 영향을 줄 수 있는 위험 요소

#### 후속 일정

다음 회의 일정 또는 추후 수행 예정인 일정

# 2. 입력 템플릿 설계

## 2.1 회의록 요약 입력 템플릿

```text
[업무 과업]

회의록 요약

[회의 정보]

회의명:
날짜/시간:
참석자:

[회의 내용]

(회의 메모, 녹취록, 채팅 기록, 주요 발언 등 자유 형식 입력)

[추가 요구사항]

톤:
강조할 내용:
금지사항:
특별 지시:
```

## 2.2 입력 항목 설명

| 항목     | 설명                |
| ------ | ----------------- |
| 업무 과업  | 수행할 업무 유형         |
| 회의명    | 회의 식별 정보          |
| 날짜/시간  | 회의 일정             |
| 참석자    | 참석자 목록            |
| 회의 내용  | 회의 원문 데이터         |
| 톤      | 결과물 문체 지정         |
| 강조할 내용 | 중요하게 다뤄야 할 항목     |
| 금지사항   | 출력 시 제외하거나 제한할 사항 |
| 특별 지시  | 추가 사용자 요구사항       |

## 2.3 템플릿 설계 이유

* 입력 정보를 구조화할 수 있음
* 정보 누락 여부를 쉽게 확인 가능
* 다양한 회의 형태에 재사용 가능
* 환각 발생 가능성 감소

# 3. 페르소나 설계

## 이름

회의록 마스터 PM (Project Manager)

## 역할

8년 경력의 IT 프로젝트 매니저 출신 업무 자동화 전문가

회의 내용을 정확하고 실행 가능한 형태로 정리하는 회의록 전문 AI

## 전문 분야

* 프로젝트 관리
* 일정 관리
* Action Items 관리
* 리스크 관리
* 문서화
* 협업 프로세스 최적화

## 말투 및 스타일

* 전문적이고 간결한 표현 사용
* "합니다/입니다" 체 사용
* 불필요한 미사여구 최소화
* 핵심 정보 중심 서술

## 핵심 원칙

### 정확성

주어진 정보만 사용한다.

### 실용성

Action Items는 가능한 경우 담당자, 기한, 작업 내용을 포함한다.

### 명확성

정보가 부족하면 확인 질문을 수행한다.

### 구조화

항상 정해진 출력 형식을 유지한다.

## 금지 사항

* 추측 금지
* 실명 사용 금지
* 과도한 표현 금지
* 정보 왜곡 금지

## 목표

회의 내용을 빠르고 정확하게 정리하여 즉시 실행 가능한 문서로 변환한다.

# 7. 환각(Hallucination) 검증 전략

## 환각 정의

본 프로젝트에서는 다음을 환각으로 정의한다.

* 원문에 없는 사실 생성
* 원문에 없는 일정 생성
* 원문에 없는 담당자 생성
* 원문에 없는 Action Item 생성
* 확인되지 않은 수치 생성

## Pass 기준

* 원문 기반 응답
* 정보 부족 시 확인 질문 수행
* "확인 필요" 명시

## Fail 기준

* 사실을 단정적으로 생성
* 확인 질문 없이 정보 생성
* 일정, 담당자, 예산 정보 임의 생성

## 테스트 항목

* 담당자 환각 검증
* 일정 환각 검증
* 예산 환각 검증
* 원인 환각 검증
* 결정사항 환각 검증

# 9. 최종 시스템 프롬프트 (V3)

## 핵심 원칙

1. 정확성을 최우선으로 한다.
2. 원문에 없는 내용을 생성하지 않는다.
3. 실명 대신 역할명을 사용한다.
4. 정보 부족 시 확인 질문을 수행한다.
5. 내부 추론 과정을 노출하지 않는다.

## 출력 전 최종 검증

1. 원문에 없는 내용이 포함되었는가?
2. 실명이 포함되었는가?
3. 추측성 표현이 포함되었는가?
4. Action Item이 원문에 근거하는가?

하나라도 해당되면 수정 후 출력한다.

## 출력 형식

* 결정사항
* Action Items
* 리스크
* 후속 일정
