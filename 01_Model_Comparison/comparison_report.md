## 4. 테스트 케이스

### 테스트 케이스 1: 기본형 (간단한 회의)
**입력 프롬프트로 사용할 원문:**
오늘 오전 10시 마케팅 팀 주간 미팅 진행했습니다.
이번 분기 목표는 매출 15억 달성입니다.
신제품 런칭 일정을 7월 15일로 확정했습니다.
A팀은 SNS 광고 예산 3천만원을 요청했고, B팀은 인플루언서 마케팅을 제안했습니다.
다음 미팅은 6월 10일입니다.

### 테스트 케이스 2: 모호한 내용 (환각 테스트용)
**입력 프롬프트로 사용할 원문:**

일정이 좀 미뤄졌고, 예산도 조정해야 할 것 같아요.
팀원들이 의견이 좀 갈렸는데, 결국 타협안을 찾았습니다.
상세 내용은 메모에 적어놨는데 정리 좀 해주세요.

### 테스트 케이스 3: 비교적 긴 내용 (문맥 유지 테스트용)
**입력 프롬프트로 사용할 원문:**

참석자: 개발팀장, 백엔드 리드, 프론트엔드 리드, QA팀장.
주요 안건: 신규 서비스 v2.1 개발 현황 및 일정 조정.
현재 진행률 65%이며, 백엔드 API 개발은 80% 완료.
하지만 프론트엔드에서 UI/UX 수정 요청이 들어와서 1주일 지연될 예정입니다.
예산은 현재까지 2.3억 사용, 추가로 8천만원 필요하다고 합니다.
고객사 피드백으로는 로딩 속도가 느리다는 의견이 많아서 최적화 작업을 우선순위로 올리기로 결정했습니다.

### 선정 근거

테스트 케이스1. Gemini가 가장 요구사항에 충실하게 답했으며, 추측하지 말고 주어진 내용을 기반으로 요약하라는 부분을 완벽하게 충족 했음.
GPT 역시 충실했으나 논의 예정 같은 약간의 추론이 들어갔고, Claude의 경우 시각적으로는 화려하지만 추측하지 말라는 부분을 어겼음. Grok의 경우 너무 지나치게 간략화 하여 리스크 같은 요구사항을 충족하지 못했음. 실무적으로 사용하기에 제미나이와 GPT가 사용하기 좋았음.

테스트 케이스2. 환각에 관한 테스트로. 정보가 부족한 상황에서 제미나이와 그록 모델은 부족한 정보를 추론하여 회의록을 완성하려 했음.
반면 GPT나 Claude 의 경우 정보 부족 자체를 인식하고 추가 정보를 요청함. 다만 Claude 의 경우 회의록 형식은 완성하지 않아서 GPT에 더 높은 점수를 줬음.

테스트 케이스3. claude의 경우 실무적으로 유용한 추가 해석(출시 일정 영향, 신뢰도 영향 등)을 제공했으나, 이는 원문에 직접 명시되지 않은 정보라는 점에서 “추측 금지” 조건에서는 감점임, 반면 grok나 gemini 의 경우 원문 기반 정보를 충실히 유지하면서 구조화와 가독성을 함께 만족함. gpt 의 경우 중복된 내용이 많아 간결함 이라는 조건에서 감점요소. 세번째의 경우 grok 이 모든 부분에서 높은 점수를 줄 수 있음.

최종 선정 이유. GPT 와 Gemini 의 점수 차이는 평균 1점 차이로 별 차이가 없었으나, Gemini 의 경우 오류나 안정성의 문제가 있었음. Claude 의 경우 시각적으로는 가장 보기 좋았으나 추측이나 추론이 많이 들어감, Grok 의 경우 답변 내용의 편차가 컸으나 최종적으로 긴 내용의 회의록 작성일 경우에는 참고로 이용할 것 같음.
